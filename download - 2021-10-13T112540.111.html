              Inheritance relationships can be defined between classes in situations where more specific child classes extend the functionality of more general parent classes. These hierarchies help us write well-organized code and can allow for code re-use.

              Overview

              Inheritance

              When youâre attempting to solve a programming problem, you will often find an existing class that does almost what you need. If you changed the class, you could end up breaking something that used to work.

              You could also create a new class and copy and paste from the old one the functionality that you want to borrow. Then, you could add the new functionality that you need. This could end up creating some confusion in the code if these two classes are similar. What would you do if you changed the functionality in the original class? Youâd likely want to track down the correlating code in the new class and change that. You can see how this would create some maintenance challenges in your code base.

              Instead of changing the old class or cutting and pasting to create a similar class, you can use inheritance to solve your problem. When you use inheritance to create your new class, you can use all the code from the old class (copying none of it) and you can also add the additional functionality you need.

              In the new class, we only need to define what we add or change from the old class. We call the original class either the parent, superclass, or base class. We call the new class the child, subclass, or derived class.

              Letâs see how this works in code:

              >>> class Bicycle():
...     pass
...
>>> class Specialized(Bicycle):
...     pass
...
>>> a_bike = Bicycle()
>>> a_specialized = Specialized()
>>>

              In the example above, a Specialized is a Bicycle. The child class is a specialization of a parent class. We created this relationship by passing the Bicycle class into the Specialized class definition. a_specialized is an instance of class Specialized but it also inherited whatever the Bicycle class can do. In this example, both these classes are useless. Letâs now fill out our code more: 

              >>> class Bicycle():
...     def exclaim(self):
...         print("I'm a Bicycle!")
...
>>> class Specialized(Bicycle):
...     pass
...
>>> a_bicycle = Bicycle()
>>> a_specialized = Specialized()
>>> a_bicycle.exclaim()
I'm a Bicycle!
>>> a_specialized.exclaim()
I'm a Bicycle!
>>>

              By setting up the parent-child relationship when we defined the classes, now any instance of the Specialized class has access to the methods on the parent class Bicycle.

              Overriding a Method

              Letâs override the exclaim method so that an instance of Specialized will tell us what it is: 

              >>> class Bicycle():
...     def exclaim(self):
...         print("I'm a Bicycle!")
...
>>> class Specialized(Bicycle):
...     def exclaim(self):
...         print("I'm a Specialized! I'm a more specialized version of a Bicycle!")
...
>>> a_bike = Bicycle()
>>> a_specialized = Specialized()
>>> a_bike.exclaim()
I'm a Bicycle!
>>> a_specialized.exclaim()
I'm a Specialized! I'm a more specialized version of a Bicycle!
>>>

              Notice that just by defining a new exclaim method on the child Specialized class, any instances of the child class will now use that method instead of the parent Bicycle classâs exclaim method.

              It is important to note here that method overriding is optional. If a child class does not override a parent method, then when the target method is called on the child class, the parent method is used. Child classes can override methods on a parent class, however, the reverse is not true. Parent classes cannot override methods on their children.

              Using super

              Youâve now seen how the child class can override a method from a parent. But, what if we needed to call the parent method?

              Letâs create a Student parent class and a Graduate child class. The Graduate class needs everything a normal Student has, but also needs to store a graduation_date. We will use inheritance to set up this relationship. 

              >>> class Student():
...     def __init__(self, name):
...         self.name = name
...
>>> class Graduate(Student):
...     def __init__(self, name, graduation_date):
...         super().__init__(name)
...         self.graduation_date = graduation_date
...
>>>

              By defining the __init__ method in Graduate, we are overriding the __init__ method in Student. But, we still want the Student portion of our child class to run its initialization, even when instantiating a Graduate. Inside the child Graduate, to access the parent Studentâs method, we use super(). Then, after running the parent initializer, we run the Graduate specific initialization behavior.

              One thing to notice is that when we called super().__init__() in the child class, we didnât have to pass along self. The self argument is passed to the parent class automatically.

              Letâs create an instance of Graduate. 

              >>> john_lennon = Graduate('John Lennon', 'December 8, 1980')
>>> john_lennon.name
'John Lennon'
>>> john_lennon.graduation_date
'December 8, 1980'
>>>

              Now that you have explored inheritance relationships, we will look at another category of class relationships - association.

              Overview

              Composition in programming is when we collect several objects together to create a new one. Composition is usually the right choice when one object is part of another object. Itâs often said that if inheritance establishes an âis-aâ relationship, then composition establishes a âhas-aâ relationship.

              To illustrate, imagine youâve created a Duck class. A duck is made up of many parts, or a duck has many parts. So, a Tail class would not inherit from the Duck class, rather a Duck would be composed of the Tail class (and many others like Bill, Eye, Foot, etc.).

              Iâm sure you are already wondering, what would this look like in Python code? Letâs define a few classes to illustrate. 

              class Bill():
    def __init__(self, description):
        self.description = description


class Tail():
    def __init__(self, length):
        self.length = length


class Duck():
    def __init__(self, bill, tail):
        self.bill = bill
        self.tail = tail

    def about(self):
        print(
            f"This duck has a {self.bill.description} and a {self.tail.length} tail.")


duck = Duck(Bill('wide orange'), Tail('long'))
duck.about()

              Notice how none of the classes have an inheritance relationship with the others. However, the Duck class expects to receive instances of the Bill and Tail classes when it is initialized. You can tell this is a composition relationship because you can accurately describe the program by stating that the Duck has a Bill and Tail.

              Aggregation

              Aggregation is almost exactly like composition. The difference is that aggregate objects can exist independently. It would be impossible for a bill to be associated with a different duck. But what if there was a Collar or Leash class? A Duck could have a collar or leash (although itâd be strange I admit), and, unlike a Bill or Tail, the collar or leash could exist independently from the Duck.

              Another way to differentiate between aggregation and composition is to consider the objectâs lifespan. If the outside object (Duck in this example) controls when the inside objects are created and destroyed (like Bill or Tail), composition is most suitable. If the related object (like Leash or Collar) can outlast the composition object (Duck in this example), then an aggregate relationship makes more sense.

              Keep in mind that composition is aggregation; aggregation is just a more general form of composition. Any composite relationship is also an aggregate relationship, but the reverse is not true.
