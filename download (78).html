      Learn

      Learn to recall the time and space complexity, the strengths and weaknesses, and the common uses of a hash table

      Overview

      Hash tables are also called hash maps, maps, unordered maps, or dictionaries. A hash table organizes data in a way that allows you to quickly look up values for a given key. You can also call them maps or hash maps because they map keys to values.

      Follow Along

      Time and Space Complexity

      Lookup

      Hash tables have fast lookups (O(1)) on average. However, in the worst case, they have slow (O(n)) lookups. The slow lookups happen when there is a hash collision (two different keys hash to the same index).

      Insert

      Hash tables have fast insertions (O(1)) on average. However, in the worst case, they have slow (O(n)) insertions. Just like with the lookups, the worst case occurs due to hash collisions.

      Delete

      Hash tables have fast deletes (O(1)) on average. However, in the worst case, they have slow (O(n)) deletions. Just like with lookups and insertions, the worst case occurs due to hash collisions.

      Space

      The space complexity of a hash table is linear (O(n)). Each key-value pair in the hash table will take up space in memory.

      Strengths

      The primary strengths of hash tables are their fast lookups. If you want a data structure that gives you O(1) lookups in the average case, hash tables are great. You can also use any hashable object as a key in a hash table, making hash tables extremely flexible.

      Weaknesses

      Although hash tables are a handy and flexible data structure, they do have some weaknesses. First, in the worst case, where your hash table has many collisions, all the operations devolve into having O(n) time complexity. Second, the keys are not stored in any order. So, if you need to find the largest or smallest key, youâd have to search through every single key to identify it. Third, similar to the previous weakness, lookups only work in one way. If you have the key, you can quickly retrieve the value. However, if you have the value and need to get the keys, that is a slow O(n) operation. Last, hash tables often use linked-lists under the hood to store data in each slot. Linked lists donât put data right next to each other in memory and therefore are not cache-friendly.

      What About Hash Collisions?

      A hash collision is when our hash function returns the same index given two different keys. Theoretically, there is no perfect hash function, though some are better than others. Because of this, any hash table implementation has to have a strategy to deal with the scenario when two different keys hash to the same index. You canât just have the hash table overwrite the already existing value.

      The most common strategy for dealing with hash collisions is not storing the values directly at an index of the hash tableâs array. Instead, the array index stores a pointer to a linked list. Each node in the linked list stores a key, value, and a pointer to the next item in the linked list.

      The above is just one of the ways to deal with hash collisions. Hopefully, you can now see why all of our hash table operations become O(n) in the worst case. What is the worst case? The worst case is when all of the keys collide at the same index in our hash table. So, if we have ten items in our hash table, all ten items are stored in one linked list at the same index of our array. That means that our hash table has the same efficiency as a linked list in the worst case.

      Overview

      class HashTable:
    """
    A hash table with `capacity` buckets
    that accepts string keys
    """

    def __init__(self, capacity):
        self.capacity = capacity  # Number of buckets in the hash table
        self.storage = [None] * capacity
        self.item_count = 0

    def get_num_slots(self):
        """
        Return the length of the list you're using to hold the hash table data. (Not the number of items stored in the hash table,
        but the number of slots in the main list.)
        One of the tests relies on this.
        """
        return len(self.storage)

    def djb2(self, key):
        """
        DJB2 hash, 32-bit
        """
        # Cast the key to a string and get bytes
        str_key = str(key).encode()

        # Start from an arbitrary large prime
        hash_value = 5381

        # Bit-shift and sum value for each character
        for b in str_key:
            hash_value = ((hash_value << 5) + hash_value) + b
            hash_value &= 0xffffffff  # DJB2 is a 32-bit hash, only keep 32 bits

        return hash_value

    def hash_index(self, key):
        """
        Take an arbitrary key and return a valid integer index within the hash table's storage capacity.
        """
        return self.djb2(key) % self.capacity

    def put(self, key, value):
        """
        Store the value with the given key.
        """


    def delete(self, key):
        """
        Remove the value stored with the given key.
        Print a warning if the key is not found.
        """


    def get(self, key):
        """
        Retrieve the value stored with the given key.
        Returns None if the key is not found.
        """
